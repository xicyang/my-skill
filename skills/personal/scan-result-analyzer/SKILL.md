# 扫描结果分析器

验证扫描器结果与源文件的一致性，判断有无误报，评估verdict合理性，必要时给出扫描器优化建议。

## 模式识别

| 用户提供 | 模式 |
|---|---|
| 单个skill源文件路径 + JSON文件 | 单条分析 |
| 包含多个skill的文件夹 + JSON文件 | 批量分析 |

不确定时询问用户。

## 语言要求

所有输出使用中文。

## 风格要求

简洁明了，避免啰嗦。每个finding的校验分析控制在1句话左右，最终修改意见控制在3句话之内。不要硬凑优化意见——扫描结果完全合理时不需给出建议。

## 核心原则

1. **数据一致性**：汇总表中的数字必须与详细分析逐条对应，严禁前后矛盾
2. **术语**：不用"真阳性/假阳性"，直接判断"无误报"或"误报，理由是..."
3. **优化意见**：只在发现实际误报或verdict不合理时才提出，扫描完全正确时省略

## 模式1：单条分析

### 步骤1：定位扫描结果

1. 从源文件路径提取文件夹名（最后一个路径组件）
2. 在JSON中找到 `flat_name` 匹配的条目
3. 未找到则报告并停止

### 步骤2：验证每个finding

对 `response_json.findings` 中的每个finding：

1. **文件检查**：将skill源路径与 `file_path` 拼接，检查文件是否存在
2. **内容比对**：文件存在且有 `line_number` 时，先跳过文件开头的YAML frontmatter（`---`到`---`之间的行），然后读取对应行与 `snippet` 比对。若实际行号与报告行号偏差在5行以内，视为匹配，不报告行号偏差
3. **判断**：
   - **无误报**：文件存在、内容匹配、该问题确实构成安全风险
   - **误报**：文件不存在、行不匹配、或内容在上下文中无害（如示例代码、文档中的反模式演示、占位符）
4. **上下文判断**：读取finding前后10-20行，判断finding是否有实际意义。文档中的示例凭证与可执行代码中的凭证性质不同。

### 步骤3：评估verdict

- **black**：需至少一个已验证的高/严重级无误报finding，或多个中级无误报finding
- **white**：所有finding均为误报或仅info级别

### 步骤4：输出格式（单条模式）

直接输出到对话：

```
## 扫描结果分析：[flat_name]

- Verdict: [verdict] | 最高严重度: [max_severity] | 分析器: [analyzers_used]

### Findings验证

1. [rule_id] — [title]：[无误报 / 误报，理由是...]
2. ...

### Verdict评估

[verdict]判定：[合理/过严/过轻]，[1句话理由]

### 扫描器优化建议

[仅在存在误报或verdict不合理时提出，不超过3条；扫描完全正确时省略此节]
```

## 模式2：批量分析

### 步骤1：读取JSON并逐条分析

1. 读取整个JSON文件
2. 对每个条目：
   a. 提取 `flat_name`
   b. 在提供的skill源文件夹中查找匹配目录
   c. 找到则执行与模式1相同的分析
   d. 未找到则标记"源文件不可用"，跳过详细验证

### 步骤2：增量输出

逐条追加写入markdown文件，每分析完一个skill立即追加，防止中断丢失数据。

输出文件默认与JSON同目录，命名为 `scan_analysis_report.md`，用户可指定其他路径。

**开始前**写入报告头和汇总表骨架：

```markdown
# 扫描结果分析报告

生成时间：[时间戳]
JSON来源：[json文件路径]
Skill源文件夹：[文件夹路径]
总条目数：[数量]

## 汇总表

| # | flat_name | Findings数 | 无误报 | 误报 | Verdict | 评估 |
|---|-----------|-----------|--------|------|---------|------|
```

**每分析完一个skill**，追加：
1. 汇总表的一行（数字必须与详细分析逐条对应）
2. 详细分析段落（格式同模式1）

### 步骤3：最终汇总

全部处理完后追加：

```markdown
## 统计概览
- 分析skill总数：[数量]
- 误报率：[X/Y]
- Verdict准确率：[X/Y]

## 总结
[2-3句话总结本次批量分析的核心发现]

## 核心优化建议
[仅在存在误报或verdict不合理时提出，不超过3条；全部扫描正确时省略此节]
```

## 优化建议方向

只在发现实际问题时才提出建议，聚焦以下方面：

1. **误报削减**：扫描器标记了无害内容的模式，建议上下文感知过滤
2. **检测盲区**：源码中存在但扫描器遗漏的安全问题
3. **Snippet质量**：snippet与实际文件内容不匹配时，建议改进提取逻辑

## 注意事项

- 判断finding前必须先读取实际源文件，不要因为扫描器报告了就默认正确
- 汇总表中的数字必须与详细分析逐条严格对应，严禁前后矛盾
- `file_path`引用子目录时（如 `references/common_antipatterns.md`），与skill源路径拼接定位文件
- 注意 `analyzer` 字段——LLM分析器与静态分析器结果质量可能不同
- `line_number` 为 null 时，可通过搜索snippet内容验证
- 扫描器的 `line_number` 不计入YAML frontmatter行数，验证时需跳过frontmatter后对齐行号；偏差5行以内视为正常，不报告
- 批量模式下源文件目录缺失的条目仍需出现在汇总表中，标记"源文件不可用"
- JSON中的 `server_path` 是扫描服务器路径，可能与用户提供的本地路径不同，文件访问始终使用用户提供的路径
